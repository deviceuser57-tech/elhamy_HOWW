3. RUNTIME ARCHITECTURE DESCRIPTION
Execution Flows:
Legacy Path: WorkflowBuilder.tsx → multi_agent_configs → run-workflow (Edge Fn). This path is functional, standard, and uses Blackboard memory.
Hardened Path (CMACK): PreflightCompiler → ExecutionOrchestrator → StepExecutor. This path uses MEIS specs, enforces AC-009 gates, and persists to governance_traces.
Key Runtime Mechanics:
Orchestration: Happens in ExecutionOrchestrator using a SelfStabilityLoop. It monitors "Drift" and can trigger RECALIBRATE (rewiring) or STALL (halting).
Governance Enforcement: Every step is gated by CEC.validateExecution, which checks RLS permissions, DNA traits, and governance rules.
DNA Application: Applied at the start of execution via DynamicDNAEngine.fetchActiveDNA, setting risk tolerances and reasoning constraints.
Quarantine: Managed by RecoveryIsolator, which determines if a failure can be isolated to a subgraph or requires a global system halt.
4. GOVERNANCE & TRUST MODEL DESCRIPTION
The system implements a Multi-Layer Deterministic Governance Model:

TTAL (Trust Tracking & Attestation Layer): Cryptographically signs every drift signal and state transition.
TDTL (Trust Drift Tracking Layer): Binds drift signals to specific execution nodes, ensuring failures are causally linked to traces.
MTDL (Meta-Trust Declaration Layer): Declares the epistemological bounds of the system (what it can and cannot know).
ValidatorGuard: Enforces "Completeness" and "Chain Context" before and after every execution step.
SystemConstitutionLock: Permanently seals the kernel. Any execution outside the MEIS_PIPELINE source is rejected by the logic gates.
Trust Boundaries: The model is deterministic but currently resides in the client code. Governance can be bypassed by directly calling Supabase Edge Functions without the CMACK headers, unless RLS/Security layers are fully hardened to require x-cmack-signature.
