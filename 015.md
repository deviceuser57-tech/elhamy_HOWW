│  │  • المدة: ثوانٍ — دقائق                                                           │   │
│  │  • الوظيفة: الاحتفاظ بالمعلومات أثناء المعالجة                                    │   │
│  │  • السعة: محدودة (15-30 عنصر)                                                     │   │
│  │  • الاضمحلال: سريع (معدل 0.1/ثانية)                                               │   │

│  │  • الارتباط: الارتباط بمعرفة موجودة يزيد القوة                                    │   │

│  ┌─────────────────────────────────────────────────────────────────────────────────┐   │

9.3 آلية النسيان

النسيان ليس فشلاً — هو ميزة ضرورية لمنع تضخم الذاكرة.
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                    FORGETTING MECHANISM — آلية النسيان                                  │
├─────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                         │
│  قواعد النسيان:                                                                        │
│                                                                                         │
│  1. Decay-Based Forgetting (النسيان بالاضمحلال)                                      │
│     • كل ذاكرة لها "قوة" (strength) تتناقص بمرور الوقت                             │
│     • معدل الاضمحلال يختلف حسب نوع الذاكرة                                           │
│     • Sensory: 0.5/ثانية                                                            │
│     • Short-Term: 0.1/دقيقة                                                         │
│     • Working: 0.01/ساعة                                                            │
│     • Episodic: 0.001/شهر                                                           │
│     • Semantic: 0 (لا تنسى — لكن تُحدَّث)                                           │
│                                                                                         │
│  2. Interference-Based Forgetting (النسيان بالتداخل)                                 │
│     • ذكريات متشابهة تتداخل وتُضعف بعضها                                             │
│     • يُحتفظ بالأقوى ويُنسى الأضعف                                                  │
│     • يُستخدم في Short-Term Memory فقط                                               │
│                                                                                         │
│  3. Purposeful Forgetting (النسيان المقصود)                                         │
│     • ذكريات خاطئة أو مُضلِّلة تُحذف فوراً                                          │
│     • ذكريات مرتبطة بقضية مغلقة تُحذف بعد فترة (بناءً على سياسة الاحتفاظ)         │
│     • ذكريات قديمة جداً وغير مفيدة تُحذف (Garbage Collection)                       │
│                                                                                         │
│  4. Consolidation-Based Forgetting (النسيان بالتثبيت)                               │
│     • عند تثبيت ذاكرة في Semantic Memory، يُحذف الأصل من Short-Term                  │
│     • التفاصيل غير المهمة تُفقد، والمعنى المستخلص يُحتفظ به                         │
│                                                                                         │
│  منع التضخم:                                                                           │
│  • حد أقصى لكل نوع ذاكرة (Soft Limit + Hard Limit)                                  │
│  • عند تجاوز الحد: تفعيل Garbage Collection                                          │
│  • أولوية الحفظ: Semantic > Episodic > Working > Short-Term > Sensory               │

11.1 المبدأ
───────────
"لا LangGraph. لا CrewAI. لا AutoGen. لا أي إطار جاهز."
C-LOS يمتلك محرك تنسيق أصلي (Native Orchestration Engine) مصمم خصيصاً
للتفاعل المعرفي بين الوكلاء — ليس مجرد جدولة مهام.

مراجعة شاملة لنظام C‑LOS

نظرة عامة

النظام C‑LOS هو إطار متكامل لإدارة المعرفة القانونية وتنسيق الوكلاء الذكيين. يتضمن مكونات أساسية:



محرك الإعادة الذاتية (Self‑Reformation Engine)

نظام الذاكرة المتعددة الطبقات (Memory Governance System)

نواة DNA (DNA Core)

محرك التنسيق المعرفي (Cognitive Orchestration Engine)

قواعد تدفق المعرفة

مستندات تعريفية، معمارية، وملخص تنفيذي

ملاحظات عامة (نقاط القوة)

فصل واضح بين التشخيص، التخطيط، والتنفيذ → يقلل من تعقّب الأخطاء ويسمح بمراجعة مستقلة.

حكمية ذاتية (Governance) مع موافقة قبل التعديل → يمنع تعديل النظام بدون إشراف.

نظام الذاكرة المتعدد الأنماط (Episodic, Semantic, Procedural, Institutional) → يعكس نماذج معرفية بشرية ويتيح تعلم مستمر.

مبادئ سيادة المعرفة (Self‑Ownership, Self‑Protection …) → توجيه واضح للخصوصية وأمن البيانات.

مستوى توثيق عالي – ملفات Markdown مفصلة مع رسومات بيانية.

نقاط الضعف / النواقص (ما يحتاج تحسين)

غياب طبقة Sandbox / بيئة محاكاة قبل تطبيق أي تغيّر على النظام الحي (مذكور في القسم 8).

آلية تصويت بسيطة في مجلس الحوكمة – لا توضح وزن الأصوات أو صلاحيات DNA Core.

عدم تصنيف أنواع الإصلاحات (مستوى 1‑4) – يصعب تقدير المخاطر.

غياب ميزانية أمان (Safety Budget) لتحديد عدد/نسبة الإصلاحات أسبوعيًا.

نقص في وثائق اختبار الوحدة (unit tests) وخطوط الأنابيب CI/CD.

لا يوجد توضيح لكيفية تكامل الواجهة الأمامية (UI) مع الـ Orchestration Engine.

قواعد تدفق الذاكرة إلى الأرشيف غير مفصلة كـ “Conflict Resolution Layer”.

توصيات التحسين (مقترحات تفصيلية)

1. إضافة Sandbox Layer

إنشاء مكوّن محاكاة يُدعى ShadowSystem يتلقى نفس المدخلات كما النظام الحي ويعيد النتائج للمقارنة قبل التنفيذ الفعلي.

دمجها بين مرحلة Planner و Approval بحيث يتم اختبار الخطة على الـ ShadowSystem أولًا.

2. تعزيز مجلس الحوكمة

تعريف أدوار واضحة (مثلاً: رئيس المجلس، مستشار أمان، ممثل DNA Core).

إضافة وزن صوتي (Weight) لكل دور؛ إعطاء DNA Core حق الـ Veto.

توثيق بروتوكول اتخاذ القرار (إجماع/أغلبية/وزن).

3. تصنيف أنواع الإصلاحات

المستوى	الوصف	مثال

1	تعديل معلمة (Parameter Change)	تغيير عتبة pLDDT للثقة

2	تعديل سير عمل (Workflow Change)	إضافة خطوة Validation في الـ Planner

3	تعديل بنية الوكيل (Agent Structure Change)	إضافة مكوّن LegalReasoner جديد

4	اقتراح دستوري (Constitutional Proposal)	تعديل مبدأ سيادة المعرفة

4. ميزانية أمان

تحديد حد أقصى للإصلاحات الأسبوعية (مثلاً: ≤2 من المستوى 3 أو ≤10 % من مكونات النظام).

إنشاء مكوّن SafetyBudgetMonitor يتحقق من الحد قبل موافقة المجلس.

5. تطوير اختبارات الوحدة & CI/CD

كتابة اختبارات لكل مكون رئيسي (Planner, Approval, Executor, Memory Layers).

إعداد خطوط أنابيب GitHub Actions أو Azure Pipelines لتشغيل الاختبارات، فحص القواعد (linters)، وإصدار الحزم.

6. توثيق واجهة المستخدم / API

إنشاء مواصفات OpenAPI للـ Orchestration Engine.

توفير دليل Front‑End يوضح كيفية استدعاء خدمات الـ Scheduler، Planner، وReformation Engine.

7. إدخال Conflict Resolution Layer في الذاكرة

إضافة مكوّن MemoryConflictResolver يحدد سياسات (Replace, Merge, ReduceConfidence, KeepBoth) عند تصادم المعرفة القديمة والجديدة.

توثيق السيناريوهات في ملف Memory_Governance.md.

8. تحسين قواعد تدفق المعرفة

توضيح قواعد تدفق إضافية مثل “لا تدفق من Extraction Gateway إلى DNA Core مباشرة”.

إضافة سجل تدفق (Audit Trail) موحد يُخزن في قاعدة بيانات PostgreSQL مع timestamps.

خطوات التنفيذ (الترتيب المنطقي)

إعداد بيئة التطوير

تفعيل git وإعداد مستودع C-LOS.

إنشاء فرع enhancement/sandbox‑layer.

إضافة Sandbox Layer

تصميم مكوّن ShadowSystem (ملف shadow_system.py).

تعديل Planner ليُرسل الخطة إلى الـ ShadowSystem أولًا.

كتابة اختبارات وحدة للتحقق من تشابه المخرجات بين ShadowSystem والنظام الحي.

تحسين مجلس الحوكمة

إضافة هيكل GovernanceCouncil في governance.py مع أدوار وأوزان.

تحديث Approval لاستخدام الأوزان وفحص حق الـ Veto لـ DNA Core.

تعريف ReformationType (Enum) ومكوّن SafetyBudgetMonitor.

إضافة فحص الميزانية قبل إعطاء موافقة المجلس.

إدخال Conflict Resolution Layer

إنشاء memory_conflict_resolver.py.

ربطه بـ Memory Governance System في تدفق البيانات.

توسيع اختبارات الوحدة وخطوط CI/CD

كتابة ملفات اختبار tests/ لكل مكوّن محدث.

إعداد GitHub Actions workflow ci.yml لتشغيل الاختبارات، lint، وتوليد تقارير.

توثيق API وواجهة المستخدم

إنشاء openapi.yaml.

كتابة دليل Front‑End frontend_guide.md.

مراجعة وتحديث قواعد تدفق المعرفة

تعديل ملف knowledge_flow_rules.md لإضافة القواعد الجديدة.

ربط الـ Audit Trail بقاعدة PostgreSQL.

مراجعة شاملة وتنفيذ

دمج جميع الفروع في main بعد مراجعة الكود (Pull Request).

تشغيل اختبارات التكامل على البيئة التجريبية.

إذا نجحت، تطبيق التغييرات على النظام الحي بعد موافقة المجلس.

خطة التحقق

اختبارات وحدة لكل مكوّن جديد (Coverage ≥ 90 %).

اختبارات تكامل: تشغيل سيناريوهات end‑to‑end على الـ ShadowSystem ثم على النظام الحي.

مراجعة يدويّة: تدقيق سجل التدفق (Audit Trail) بعد كل تعديل.

تحليل أمان: تشغيل فحص Static Analysis (Bandit) واختبار اختراق بسيط (OWASP‑ZAP).

مراجعة مستندات: تدقيق ملفات Markdown لضمان اتساق المصطلحات وروابط الملفات.
